# Code Review Checklist

各項目には `[CR-XX-NNN]` 形式の checklist_id を付与している。
レビュー時に指摘する場合はこの ID を併記すること（F-001: レビュー判断ヒント機能の照合キー）。

## 必須チェック項目（全作成物共通）

### コード品質
- [ ] [CR-Q-001] 関数の責務が単一か（単一責任原則・1関数1役割）
- [ ] [CR-Q-002] 関数の長さが適切か（目安50行以内。超える場合は分割を検討）
- [ ] [CR-Q-003] ネストの深さが適切か(目安3段階以内。深い場合は早期リターンや関数分割を検討)
- [ ] [CR-Q-004] マジックナンバー・マジック文字列が定数化されているか
- [ ] [CR-Q-005] デッドコード(到達不能コード・未使用変数・未使用関数)が残っていないか
- [ ] [CR-Q-006] コメントアウトされたコードが残っていないか(削除またはチケット化すること)
- [ ] [CR-Q-007] 不要なデバッグ出力(console.log / print等)が残っていないか

### 命名規則
- [ ] [CR-N-001] 変数・関数名が意図を表しているか（何をするかが名前からわかるか）
- [ ] [CR-N-002] ブール値の変数名が is / has / can / should で始まっているか
- [ ] [CR-N-003] 略語を使わずに書かれているか（tmp → temporary、btn → button等）
- [ ] [CR-N-004] 一貫した命名規則が守られているか（camelCase / snake_case等がコードベース全体で統一されているか）

### エラーハンドリング
- [ ] [CR-E-001] 全ての例外・エラーが適切にハンドリングされているか
- [ ] [CR-E-002] エラーを握りつぶしていないか（空のcatch / 無言のfailが存在しないか）
- [ ] [CR-E-003] エラー発生時にリソース（ファイル・DB接続・ネットワーク）が正しく解放されているか
- [ ] [CR-E-004] ユーザーへのエラーメッセージが適切か（内部情報を含まないか・わかりやすいか）
- [ ] [CR-E-005] エラーの伝播方法が適切か（再スロー・ラップ・変換が適切か）

### ログ
- [ ] [CR-L-001] ログ出力が適切なレベル（DEBUG / INFO / WARN / ERROR）で行われているか
- [ ] [CR-L-002] 個人情報・秘密情報がログに出力されていないか
- [ ] [CR-L-003] トラブルシューティングに十分な情報がログに含まれているか
- [ ] [CR-L-004] ログが過剰でないか（ループ内での大量出力等）

### テスト
- [ ] [CR-T-001] 新機能・修正に対してテストが追加・更新されているか
- [ ] [CR-T-002] 既存テストが全て通過するか
- [ ] [CR-T-003] テスト名から何をテストしているか明確にわかるか
- [ ] [CR-T-004] テストが実装の内部ロジックに依存しすぎていないか（振る舞いをテストしているか）

### 型安全性
- [ ] [CR-CT-001] 型アノテーション・型宣言が適切に付いているか（TypeScript / Python等）
- [ ] [CR-CT-002] `any` 型や型アサションの多用がないか（あれば理由がコメントで記載されているか）
- [ ] [CR-CT-003] null / undefined の扱いが安全か（null チェック・Optional チェーニング等）

### 保守性
- [ ] [CR-M-001] 重複コードがないか（DRY原則）
- [ ] [CR-M-002] 変更に強い設計か（マジックナンバー・ハードコードされた設定がないか）
- [ ] [CR-M-003] 複雑なロジックに説明コメントが付いているか
- [ ] [CR-M-004] 循環依存が発生していないか

### 並行処理・スレッド安全性
- [ ] [CR-CC-001] 共有リソースへのアクセスが適切に排他制御されているか（ロック・ミューテックス・セマフォ等）
- [ ] [CR-CC-002] 競合状態（Race Condition）が発生しないか
- [ ] [CR-CC-003] デッドロックが発生しないか（ロック取得順序が一定か）
- [ ] [CR-CC-004] 非同期処理のキャンセル・タイムアウトが適切に実装されているか

## 推奨チェック項目（全作成物共通）
- [ ] [CR-R-001] 依存ライブラリの追加が妥当か（既存ライブラリで代替できないか）
- [ ] [CR-R-002] ドキュメント（README・API仕様等）が更新されているか
- [ ] [CR-R-003] コードの変更がアーキテクチャ方針と一致しているか

## 作成物別チェック項目

### Web API / バックエンド向け
- [ ] [CR-WB-001] N+1クエリが発生していないか（ループ内でのDB呼び出し等）
- [ ] [CR-WB-002] 不要なデータを取得していないか（SELECT * の多用・不要なJOIN等）
- [ ] [CR-WB-003] トランザクション境界が適切か（複数テーブル更新時の整合性）
- [ ] [CR-WB-004] ページネーション・件数制限が実装されているか（大量データの一括取得がないか）
- [ ] [CR-WB-005] レスポンスに不要なフィールドが含まれていないか
- [ ] [CR-WB-006] 非同期処理が適切に実装されているか（Promise / async-await の正しい使用）
- [ ] [CR-WB-007] タイムアウトが設定されているか（外部API呼び出し・DB接続等）

### フロントエンド向け
- [ ] [CR-FE-001] 不要な再レンダリングが発生していないか（メモ化・依存配列の設定等）
- [ ] [CR-FE-002] メモリリークが発生していないか（イベントリスナー・タイマーの解放）
- [ ] [CR-FE-003] 大量データのレンダリングに仮想スクロール等の対策があるか
- [ ] [CR-FE-004] アクセシビリティ（aria属性・キーボード操作・コントラスト）が考慮されているか
- [ ] [CR-FE-005] 国際化（i18n）対応が必要な場合に対応されているか

### バッチ処理 / 非同期処理向け
- [ ] [CR-BC-001] 冪等性が担保されているか（同じ処理を複数回実行しても問題ないか）
- [ ] [CR-BC-002] 処理の途中失敗時にリカバリできるか（再実行・チェックポイント）
- [ ] [CR-BC-003] メモリ使用量が適切か（大量データを一括ロードしていないか・ストリーム処理等）
- [ ] [CR-BC-004] キューの詰まり・バックプレッシャーへの対策があるか

### ライブラリ / SDK向け
- [ ] [CR-LIB-001] 公開APIのインターフェースが安定しているか（破壊的変更がないか）
- [ ] [CR-LIB-002] 破壊的変更がある場合にバージョンが適切に上がっているか（セマンティックバージョニング）
- [ ] [CR-LIB-003] 公開APIにドキュメント（型定義・JSDoc / docstring等）が付いているか
- [ ] [CR-LIB-004] ライブラリ利用者の環境を壊さないか（グローバル変数の汚染・副作用等）
