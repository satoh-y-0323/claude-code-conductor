# NUL 境界 lint — 宣言マーカーの規約

このドキュメントは、C3 配布物に含まれるコードから見かける
`# nul-boundary: allow(<理由>)` マーカーの意味と使い方を説明します。

---

## マーカーの意味

### 標準形

```python
# nul-boundary: allow(なぜこの join が NUL 区切りでないのか)
result = "\n".join(items)
```

**意味**: この `str.join()` 呼び出しが `"\n"` や `", "` などの NUL 以外を区切りにしているのは、
意図的かつ妥当である、ということを宣言しています。

### 語彙は 1 種のみ

マーカーの形は **`allow(<理由>)` の 1 種だけ**です。

- ✅ `allow(人間可読な出力なので改行を維持)` — OK
- ❌ `human(...)` / `safe(...)` / `fixed(...)` / `legacy(...)` — これらは使いません

理由は、**マーカーは「意図的に NUL ではない」以上のことを主張しない**ためです。
真偽を機械が検査できない分類語を足すと、設計と実装が乖離した瞬間に嘘になります。
中立な 1 語に統一することで、その穴を消しています。

### 理由は 5 文字以上

理由部分（括弧の中身）は、**`strip()` 後に 5 文字以上**である必要があります。

- ✅ `allow(display output)` — 5文字
- ✅ `allow(  永続ファイルの後方互換性を保つ  )` — strip 後で十分
- ❌ `allow()` / `allow(  )` — 理由がない
- ❌ `allow(ok)` — 2文字では不足

この下限は「honor system の最小限」です。一言では説明できない程度の具体性を強制します。

---

## 配置規則

マーカーは、対象となる `join` 呼び出しに結び付けられます。その結び付けは以下の順序で行われます。

### 規則 1: 同一行（行末）

マーカーが行 `L` にあり、その行に `join` 呼び出しが含まれるなら、そこに結び付きます。

```python
result = "\n".join(items)  # nul-boundary: allow(説明)
```

これが最優先の配置方法です。

### 規則 2: 直前行

マーカーが行 `L` にあり、規則 1 が成立しない場合のみ、
その「直前行 `L+1`」に `join` があれば結び付きます。

```python
# nul-boundary: allow(説明)
result = "\n".join(items)
```

**重要**: 「直前」は厳密に `L+1` です。間に別の行が 1 行でも挟まると、結び付きません。

```python
# nul-boundary: allow(説明)
# このコメント行が間に挟まってしまった
result = "\n".join(items)     # ← この join は宣言で保護されていません
```

### 規則 3: 複数対象の場合は無効

マーカーが複数の `join` に結び付こうとする場合、**その宣言はどの対象も保護しません** (fail-closed)。

```python
x = "\n".join(items1)
y = "\n".join(items2)  # nul-boundary: allow(説明)
# ← 上下両方に結び付く可能性があるため、この宣言は無効
```

この場合、**両方** の行に宣言を書く必要があります：

```python
x = "\n".join(items1)  # nul-boundary: allow(説明)
y = "\n".join(items2)  # nul-boundary: allow(説明)
```

### 特例: f-string 内の `join`

f-string 内に `join` がある場合、マーカーを f-string の内側に置くことはできません。
この場合、**包含する文の どこか**（例えば末尾行の行末）にマーカーを置きます：

```python
result = (f"header\n"
          f"{items.join(x)}\n"
          f"footer")  # nul-boundary: allow(説明)
```

マーカーの結び付け判定では、f-string 内の `join` の行範囲を「包含する文の行範囲」に置き換えて処理します。

## なぜ NUL 境界則があるのか

C3 で「同じ穴を 6 回開けた」経験があります：

| # | 例 | 内容 |
|---|---|---|
| 1 | ポーセレン CLI 出力の処理 | `\n` 結合による境界偽装 |
| 2 | tracked ブロック | `-z` で NUL 取得したものを `\n` で再結合 |
| 3 | `format_ignored_manifest` | tracked を NUL 化しても manifest は `\n` のまま |
| 4 | 設定の指紋化 | 同型の穴が繰り返す |
| 5 | pytest 設定ハッシュ | 「軽量」な実装が同じ完全性を要求される |
| 6 | manifest の曖昧区切り | 2 スペース区切り＝同型の脅威 |

**3 回を超えたら個別に直すのをやめて、ルール化＋機械検査化する潮時** という教訓です。

機械可読なデータ（ログ行・ハッシュ入力・パス一覧など）がパイプラインを流れる場合：

- **区切りは NUL で切る**（`"\x00"`・1 文字ちょうど）
- **`\n` は人間可読出力にのみ使う**（表示・ユーザーへのメッセージ）

この分離により、外部由来の値に改行が混ざっても、機械可読側は境界を偽装されません。

---

## 既知の限界

以下の 7 項目は機械検査の対象外です。

| ID | 限界 | 影響 |
|---|---|---|
| L-1 | `+` 演算子による `\n` 連結を検出しない | 例: `a + "\n" + b` |
| L-2 | f-string / `%` 書式による組み立てを検出しない | 行集合の構築に f-string 使用 |
| L-3 | `split` 側の非対称を検出しない | NUL で結合し `\n` で分割する場合など |
| L-4 | **宣言の内容の真偽は検査しない** | `allow(...)` の理由に嘘を書けば通る |
| L-5 | NUL 区切りの場合、**要素側に NUL が含まれない**ことは検査しない | 外部由来値に NUL が混ぜば偽装は成立 |
| L-6 | 走査対象は 4 ディレクトリに限る | `.dev/hooks` / `scripts` / `tests` は対象外 |
| L-7 | **`allow` で `\n` を維持した箇所、要素側に改行が混入しないことは保証しない** | `\n` 区切りを保つ以上、要素に改行が入れば境界偽装は成立 |

このうち **L-4** と **L-7** が重要です：

### L-4: 理由の真偽は検査しない

例えば：

```python
# nul-boundary: allow(後方互換性が要る永続形式なので)
# → しかし実は一度限りの出力（互換性は不要）
result = "\n".join(items)
```

こういった嘘は機械では検出できません。
**宣言の妥当性は E フェーズのレビューと `git blame` で担保します**。

### L-7: `\n` 維持時の要素側改行

`allow` で `\n` 区切りを維持した箇所については、
要素（区切られる側）に改行が入らないことは保証しません。

例えば：

```python
# nul-boundary: allow(状態ファイルなので人間可読性のため \n を維持)
lines = log_entries  # ← このリスト要素が実は複数行テキストなら？
result = "\n".join(lines)
```

要素側での改行混入を防ぐのは呼び出し元の責務です。

## 理由文の書き方

理由には「**なぜ NUL にできない／する必要がないか**」を具体的に書きます。
書式は**自由記述**（ラベルの種別分けは不要）です。

### よい例

- `allow(人間が直接編集するファイルなので改行ベース)` — 用途が明確
- `allow(永続形式で過去バージョンとの互換性が必須)` — 制約の根拠がある
- `allow(正規表現の `\n` 境界に依存した処理が下流にある)` — 外部制約を示唆
- `allow(status/commit メッセージは`git`が改行で期待するため)` — 標準仕様に依存

### 避けるべき書き方

- `allow(human)` — 理由でなく分類語（機械検査できない）
- `allow(safe)` — 同上
- `allow(ok)` — 説明が短すぎる

## 検査は配布元でのみ

この `nul-boundary` 検査（`tests/test_nul_boundary_lint.py`）は、
C3 の配布元リポジトリでのみ実行されます。

C3 を配布先で `pip install c3-***.whl` でインストールしたユーザーは：

- マーカー `# nul-boundary: allow(...)` は、コード内のコメントとして見えます
- **検査ツール自体は wheel に含まれません** （`tests/` は配布対象外）

ユーザーが C3 をカスタマイズして新しい `join` を追加した場合：

1. このドキュメントを参考にして、**マーカーの配置規則に従って**宣言を書いてください
2. **理由は 5 文字以上、具体的に書いてください**
3. 配布元へ PR を送る際は、配布元の CI で `tests/test_nul_boundary_lint.py` が走り、
   テスト出力で確認できます
