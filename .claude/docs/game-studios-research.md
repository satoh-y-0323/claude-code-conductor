# Claude Code Game Studios 調査メモ

調査日: 2026-04-27
参照元: https://github.com/Donchitos/Claude-Code-Game-Studios
目的: C3 設計の参考フレームワークとして調査

---

## 1. きっかけと問題意識

Clade が機能追加を重ねるにつれ、以下の問題が蓄積してきた:

- `core.md` が7種類の関心事の混在する「何でも屋」ファイルになっている
- 親Claudeとサブエージェントの責務境界が3箇所（commands/・core.md・skills/agents/）に分散
- `rules/local.md` が性質の異なるルールの時系列積み重ね
- 「Skillとは何か」の定義が曖昧で、置き場所の判断基準がない

根本的な問題: **「各ファイルタイプの意味」が定義されていないため、追加のたびに迷いと混在が生じている。**

## 2. Claude Code Game Studios フレームワーク解析

### 2-1. ディレクトリ構造の概要

```
.claude/
├── agents/       # 60個のエージェント定義（3階層）
├── rules/        # 11個のパスベースルール
├── skills/       # 103個のスキル（各ディレクトリに SKILL.md）
├── hooks/        # 13個の自動検証フック（sh スクリプト）
├── docs/         # 17個のリファレンスドキュメント（人間向け）
└── agent-memory/ # エージェント間共有メモリ
```

### 2-2. 各ファイルタイプの「意味の定義」

| タイプ | 定義 | 主な読者 |
|---|---|---|
| **Agent** | 「私は誰か・何ができるか・誰と話すか」の定義 | サブエージェント自身 |
| **Rule** | ファイルパスに基づいて**自動注入**される背景知識 | エージェント（自動） |
| **Skill** | マルチフェーズのオーケストレーション手順 | 親Claude |
| **Command** | ユーザーのエントリーポイント | 人間・親Claude |
| **Docs** | 人間が読むリファレンス | 人間 |

### 2-3. エージェント定義のフォーマット（C3 が採用）

```markdown
# [Role Title]

## Core Mandate
[1-2文で役割の本質]

## Key Scope
✅ You Own: [担当すること]
❌ You Do Not Own: [担当しないこと]

## Operational Workflow
[Before / During / After の3段階手順]

## Tools & Constraints

## Related Agents
[上流 / 下流 / ピア]
```

### 2-4. AskUserQuestion ツールの使い方（v0.2.0 以降）

**Explain → Capture パターン:**
1. エージェントが分析・推論を文章で説明する
2. AskUserQuestion ツールで選択肢を構造化して提示する
3. ユーザーが選ぶ
4. エージェントが続行する

```json
{
  "questions": [{
    "question": "...",
    "options": [
      { "label": "...", "description": "..." }
    ],
    "multiSelect": false
  }]
}
```

26ファイル（14エージェント・8スキル・3プロトコルテンプレート）に統合されている。

### 2-5. エージェントの起動方式

- **Task ツール（旧名）= Agent ツール（現名）**: 仕様は同じ、名前が変わっただけ
- Game Studios のインタラクティブなエージェントは `/command` でスラッシュコマンドとして起動する
  → 親 Claude がそのペルソナを採用して会話する（サブエージェントではない）
- サブエージェント（Agent ツール）は対話が不要な処理タスクに使う

## 3. Clade と Game Studios の根本的な違い

| | Game Studios | Clade / C3 |
|---|---|---|
| 用途 | プロダクト（ゲーム）を作るフレームワーク | 汎用メタツール |
| コンテキスト軸 | どのファイルパスで作業するか | ワークフローのどのフェーズか |
| ルール注入 | パスベース自動注入 | フェーズ/ロールベース |

→ パスベースルールはそのまま転用できない。C3 の commands がフェーズごとのコンテキストを与える現在の仕組みは正しい方向性。

## 4. C3 への転用まとめ

| Game Studios の特徴 | C3 への転用 |
|---|---|
| エージェント定義フォーマット（✅/❌スコープ） | agents/*.md の構造化 ✅ 採用 |
| Skill = オーケストレーション手順 | skills/ の意味再定義 ✅ 採用 |
| Docs = 人間向けリファレンス分離 | .claude/docs/ ✅ 採用 |
| AskUserQuestion (Explain→Capture) | 全選択肢で採用 ✅ 採用 |
| ルール = 背景知識（指示ではない） | rules/ の役割再定義 ✅ 採用 |
| パスベースルール自動注入 | ❌ 汎用ツールなので不採用 |
| ドメイン特化エージェント60個 | ❌ 汎用なので少数で対応 |
