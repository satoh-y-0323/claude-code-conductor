# 自律モード オンボーディング

C3 の「自律モード（autonomous mode）」を初めて使う人向けの導入ガイド。
**手順と心構え**だけを書く。数値・条件・文法の厳密な定義は規約 SSOT である
`.claude/skills/autonomous-mode/SKILL.md` を参照すること（本文書は規約を再定義しない）。

> このガイドは `.claude/docs/` 配下の人間向けリファレンス。エージェントが実行時に読む必要はない。

---

## ① 自律モードとは

自律モードは、C3 標準ワークフローの承認ゲート（レポート承認・テスト承認・findings 分岐など）を、
ユーザーが承認した**委任プラン**に従って客観条件で自動進行させる運転モードである。

- **既定は常に HITL（Human In The Loop）**。何もしなければ従来どおり各フェーズで承認を求める。
- 有効化には**二重の opt-in** が必要:
  1. プランモードで「資源同意節を含む委任プラン」をユーザーが承認する
  2. その承認直後に、親 Claude がセッションファイルへ**モード行**を挿入する
- opt-in なしに挙動が変わることはない（誤発動ゼロを最優先に設計されている）。
  ただしこれは文書規約レベルの不変則であり、モード行の挿入自体がセレモニー経由であることを機械検証する仕組みではない
  （既知の限界は SKILL.md「モード行挿入の前提条件（不変則）」を参照）。

---

## ② 正しい始め方

**プランモードを先に通すのが唯一の正しい入口**。順序を守ること。

1. **プランモードで委任プランを作る**。プランには固定様式の `## 資源同意` 節を含める
   （ハード上限・起動体数の見込み・「承認をもって資源消費に同意」の 3 点。様式は SKILL.md の
   「委任プランの必須節（資源同意）」を参照）。
2. **プランを承認する**。この承認が「資源消費への同意」の成立点になる。
3. 承認直後に、親 Claude が `現在地:` 行の直後へモード行を挿入する。初期形は
   `モード: 自律 plan=<委任プランの絶対パス> cycles=C-3/0,E/0`（`cycles=` は opt-in 時に発効し初期値は両ゲート 0）。

**よくある誤り**:

- **`/start` を先に叩くのは誤り**。標準ワークフローを HITL で走らせてしまい、自律モードにはならない。
  自律運転は「プランモード承認 → モード行挿入」を経てから、init-session（セッション交代・コンパクト後）が
  モード行を読んで継続する形で成立する。
- 資源同意節を欠くプランで opt-in を成立させてはならない（欠落時は人間へ確認＝関所 2）。

## ③ 質問が来るのは設計どおり

自律運転中でも人間に質問が返ってくることがある。これは**故障ではなく設計どおりの安全装置**である。
以下の 2 類型（人間の関所）は自律モードでも自動選択せず、必ず人間が判断する。

| 類型 | 具体例 | なぜ人間のままか |
|---|---|---|
| **非可逆操作** | git commit / push / tag / release・ファイル削除 | 取り消せない操作は最後の人間判断を残す（検証用ビルド足場の作成・後始末は含まない） |
| **情報不足の質問** | task-question（このタスクに不足する情報）／ axis-question（以後の全タスクに効く方針） | 情報の欠落は自動では埋められない |

加えて、E フェーズ最終承認後の**引き継ぎバックログ照合**と**コミット提案**はコミット直前＝人間の関所領域であり、
自律でも AskUserQuestion を出して自動選択しない。

「質問が来た＝暴走していない」と受け止めてよい。

## ④ 止まり方（上限到達時の流れ）

自律運転は無限に回らない。C-3 監査・E レビューにはサイクル上限があり、到達すると次の流れで止まる。

1. **上限到達**（例: C-3 が上限サイクルまで回っても High/Medium が 0 に収束しない）
2. **状態保全**: ワーキングツリー・レポート群をそのまま残す（調査可能な形で保全）
3. **報告**: 収束または打ち切りで初めてまとめて人間へ報告する（逐一停止はしない）
4. **裁定**: 人間が続行・方針変更・打ち切りを判断する
5. **再開**: 裁定を受けて運転を継続、または HITL へ復帰する

実例（B1 実走）: E→C 差し戻し後に同一カウンタで早期に上限へ到達し、運転が止まった事例がある。
この経験から「差し戻しごとに小上限の別枠を付与する」規定が入った（詳細と数値は SKILL.md 参照）。
止まること自体は異常ではなく、上限に守られて安全側に倒れた結果である。

## ⑤ C-3 収束 ≠ 欠陥ゼロ

**C-3 監査が収束しても、それは「欠陥がゼロになった」ことを意味しない。**
C-3 の収束条件が保証する射程は、**その時点の計画の健全性**に限られる。

- **実測の裏付け**: C-3 が収束（High/Medium 0）した後の E レビューで、実装バグが実際に発見された事例がある。
  収束は「設計・計画レベルで High/Medium の指摘が消えた」ことであって、実装が正しいことの証明ではない。
- **構造的な限界**: C-3 を担う design-critic は **Bash を持たず、コードを実行して検証できない**
  （静的な読解のみ）。エスケープ処理・パーサ・状態機械のような組み合わせロジックの欠陥は、
  静的読解では原理的に取りこぼしうる。
- **最後の網は E の実行検証**: テスト実行を伴う E フェーズ（およびテスト全緑を客観条件とする D）が、
  実装バグに対する最後の防波堤になる。C-3 収束を「もう安全」と読み替えないこと。

## ⑥ 従量課金ユーザー向け注記

`## 資源同意` 節への承認は、**費用への同意を兼ねる**。

- **定額プラン**では、資源同意は主に「残枠（サイクル回数）の消費」を意味する。
- **従量課金（API 等）**では、資源同意は**費用の発生への同意**を意味する。
- **重要**: ハード上限が担保するのは**サイクル回数**であって費用ではない。
  D のタスク数はプラン依存で可変、stuck 時は systematic-debugger が加算され、E→C 差し戻しごとに D 相当が再走する。
  **費用の上界は規約では担保されない**。従量課金で使う場合は、この点を理解した上で opt-in すること。
